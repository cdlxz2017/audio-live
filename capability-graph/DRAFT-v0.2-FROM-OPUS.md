# 卓越执行框架 v0.2（Opus 评审改进版）

> 评审者：Claude Opus  
> 日期：2026-04-19

## 核心原则
- 简单任务快速通过，复杂任务严格把关
- 每步可验证，失败可回退
- 知识自动沉淀，不靠人工维护

## 流程总览

收到任务 → 分级 → 按级别走对应路径

### Step 0：任务分级（必须首先执行）

判断维度：
- 可逆性（能撤销吗？）
- 影响范围（几个系统？几个用户？）
- 确定性（方案明确吗？）
- 历史经验（做过类似的吗？）

分级结果：
| 级别 | 特征 | 路径 |
|------|------|------|
| Trivial | 可逆、单系统、方案明确 | 直接执行→简要记录 |
| Standard | 低风险、有参考经验 | 简化流程（跳过多方案） |
| Complex | 不可逆/多系统/无先例 | 完整流程 |
| Dangerous | 涉及数据删除/生产环境/外部通信 | 完整流程 + 必须呈报主人 |

### Trivial 路径
执行 → 验证 → 完成（不建库、不写 Thread）

### Standard 路径

1. **检索**：并行查询（能力图谱 + 副脑 + 主脑），取最佳匹配
   - 完全匹配（>0.7）→ 复用，检查适用性
   - 部分匹配（0.5-0.7）→ 作为参考
   - 无匹配 → 按 Complex 处理
2. **适用性检查**：依赖版本？环境变化？上次有坑吗？
   - 通过 → 继续
   - 不通过 → 降级为 Complex
3. **工具确认**：查工具卡片，确认能力边界
4. **风险扫描**：对照 risk-patterns 快速检查
   - 命中已知风险模式 → 调取对应规避方案
   - 未命中 → 继续
5. **执行**：分步执行，每步验证
   - 失败 → 回退 → 记录失败原因 → 决定重试(≤2次)/上报/放弃
6. **验证**：对照预定义成功标准检查
7. **记录**：更新 Thread，有新知识则建卡片

### Complex 路径

1. **问题定义**：明确问题、目标、约束、成功标准
2. **任务拆解**：是否需要拆分为子任务？
   - 是 → 每个子任务独立分级和执行
   - 否 → 继续
3. **检索参考**：同 Standard Step 1，但侧重收集参考而非直接复用
4. **方案设计**：基于参考设计方案，评估替代方案（仅在有意义时）
5. **决策**：
   - 可逆 + 低风险 → 自行决定，事后汇报
   - 不可逆 / 高风险 → 呈报主人，等待确认
6. **工具确认**：查工具卡片
7. **风险评估**（逐项检查）：
   - [ ] 涉及 DELETE/DROP/TRUNCATE？→ 备份先行
   - [ ] 需要重启服务？→ 确认影响范围和窗口
   - [ ] 依赖外部服务？→ 确认可用性和超时方案
   - [ ] 能回退吗？→ 写出具体回退命令
   - [ ] 涉及权限变更？→ 最小权限原则
8. **执行**：分步执行，每步验证，失败触发回退
9. **验证**：对照成功标准
10. **沉淀**：
    - 新建/更新 Thread
    - 失败模式 → risk-patterns
    - 新工具用法 → tools 卡片
    - 踩坑经验 → pitfalls

### 异常处理（所有路径通用）

- 工具无响应 → 等待 30s → 重试 1 次 → 上报
- 查询无结果 → 降级为更宽泛的搜索 → 仍无结果则按无先例处理
- 执行超时 → 检查进程状态 → 决定等待/终止/上报
- 连续失败 2 次 → 停止重试，上报主人，附带失败日志

### 自主决策边界（硬规则）

**可以自行决定：**
- 读取任何文件
- 工作区内的代码修改
- Docker 容器重启
- Git commit（不 push）

**必须呈报：**
- 数据库 schema 变更
- 生产环境操作
- 发送邮件/消息
- Git push
- 删除数据
- 安装系统级依赖
