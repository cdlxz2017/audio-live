# 系统修改与故障处理 SOP（标准操作流程）

> **版本**: v1.1  
> **创建时间**: 2026-04-09  
> **最后更新**: 2026-04-09  
> **状态**: ✅ 全局强制执行 — 任何系统问题必须走此流程

---

## 📋 流程概览

```
┌─────────────────────────────────────────────────────────┐
│  1. 寻找根本原因    →    2. 查阅技术手册                │
│         ↓                          ↓                    │
│  4. 评估影响范围    ←    3. 核查关联系统               │
│         ↓                                               │
│  5. 制定解决方案    →    6. 大模型交叉审核             │
│         ↓                                               │
│  7. 提交用户决策    →    8. 执行（如批准）            │
│                          ↓                              │
│                    ⚠️ 回滚方案（备用）                  │
└─────────────────────────────────────────────────────────┘
```

---

## 步骤 1：寻找根本原因

**目标**：从源头定位问题，而非处理表象

| 检查项 | 说明 |
|--------|------|
| 错误溯源 | 找到问题最初发生的位置/时间/触发条件 |
| 故障链条 | 理解问题如何传导到当前症状 |
| 可复现性 | 确认问题是否可稳定复现 |
| 相关日志 | 收集错误日志、时间线、最近变更 |

**操作**：
```bash
# 查看相关日志
journalctl -u [服务名] --since "1 hour ago" | grep -i error

# 查看 PM2 日志
pm2 logs [进程名] --lines 100 --nostream

# 检查系统状态
openclaw gateway status
pm2 list
```

**输出要求**：用一句话描述根本原因，例如：
> "问题根因：数据库连接池耗尽，导致新请求无法获取连接"

---

## 步骤 2：查阅技术手册

**目标**：理解相关技术栈的正确行为和配置方式

| 检查项 | 说明 |
|--------|------|
| 官方文档 | 查看官方技术文档/配置手册 |
| 现有 SOP | 查看是否有相关操作 SOP |
| 配置参数 | 确认正确参数取值范围和含义 |
| 版本兼容 | 确认各组件版本兼容性 |

**操作**：
```bash
# 查看 OpenClaw 配置文档
ls /home/ai/.npm-global/lib/node_modules/openclaw/docs/

# 查看项目技术文档
find /home/ai/.openclaw/workspace -name "*.md" -path "*/docs/*" 2>/dev/null

# 查看相关 SOP
ls /home/ai/.openclaw/workspace/SOP-*.md
```

**输出要求**：列出关键参考文档和核心结论

---

## 步骤 3：核查关联系统

**目标**：确认修改可能影响的所有上下游系统

| 检查项 | 说明 |
|--------|------|
| 依赖服务 | 该服务依赖哪些其他服务 |
| 被依赖服务 | 哪些服务依赖该服务 |
| 共享资源 | 数据库、Redis、文件系统等共享资源 |
| 配置关联 | 哪些配置文件存在关联 |

**操作**：
```bash
# 查看服务依赖关系
cat /home/ai/.openclaw/workspace/[项目]/

# 检查端口占用
netstat -tlnp | grep [端口]

# 检查相关 PM2 进程
pm2 list
```

**输出要求**：列出受影响系统清单，例如：
> - 受影响上游：member-service (3002)、auth-service (3004)
> - 受影响下游：karma-service (3007) 依赖 member-service
> - 共享资源：PostgreSQL (5432)、Redis (6379)

---

## 步骤 4：评估影响范围

**目标**：量化评估各系统的受影响程度

| 影响等级 | 说明 | 示例 |
|----------|------|------|
| 🔴 严重 | 核心服务中断，功能完全不可用 | Gateway 无法启动、数据库连接失败 |
| 🟠 较高 | 部分功能异常，局部受影响 | 某个 API 响应慢、某页面报错 |
| 🟡 中等 | 非核心功能受损 | 某个非关键按钮失效 |
| 🟢 轻微 | 基本无影响 | 日志格式变化、界面微调 |

**输出要求**：
```
影响评估报告
================
[服务名称]：[影响等级] — [具体影响描述]
影响时长：预估 [X] 小时
回滚难度：[简单/中等/复杂]
```

---

## 步骤 5：制定解决方案

**目标**：给出可执行的具体方案

| 检查项 | 说明 |
|--------|------|
| 方案完整性 | 包含具体命令/代码/配置变更 |
| 风险控制 | 每步有对应的回滚/中止机制 |
| 执行顺序 | 明确操作先后顺序 |
| 验证方法 | 每步执行后如何验证成功 |

**输出要求**：
```
解决方案
================
方案 A：[简述]
- 执行步骤：
  1. [命令/操作]
  2. [命令/操作]
- 风险：[高/中/低]
- 回滚方式：[简述]
- 预期结果：[描述]

方案 B：（备选）
[同上结构]
```

---

## 步骤 6：大模型交叉审核

**目标**：将解决方案交由其他大模型审查，得出最优解

| 检查项 | 说明 |
|--------|------|
| 审核模型 | 使用不同于主模型的另一个模型 |
| 审核要点 | 安全性、完整性、影响最小化 |
| 优化建议 | 是否可进一步减少影响 |

**操作**：
```bash
# 使用第二个模型审核解决方案
# 将解决方案作为 prompt 提交给备用模型
# 要求：指出潜在风险、建议改进、确认最优
```

**输出要求**：
```
交叉审核结论
================
审核模型：[模型名称]
安全性评估：[安全/有风险，描述]
完整性评估：[完整/有遗漏，描述]
优化建议：[如有]
最优方案确认：[方案 A/B/C + 理由]
```

║  影响范围：                                                  ║
║  • [系统A]：[影响等级] — [描述]                              ║
║  • [系统B]：[影响等级] — [描述]                              ║
║                                                              ║
║  推荐方案：[方案名称]                                         ║

## 步骤 8：执行与验证

**触发条件**：用户明确回复"执行"或"确认"

| 检查项 | 说明 |
|--------|------|
| 逐步执行 | 严格按照方案顺序执行 |
| 每步验证 | 每步完成后立即验证 |
| 异常停止 | 发现异常立即暂停，触发回滚 |

**操作**：
```bash
# 按步骤执行，每步后验证
[命令1] && echo "✅ 步骤1完成" || { echo "❌ 步骤1失败，停止执行"; exit 1; }

# 验证命令
[验证命令]

# 继续下一步...
```

## ⚠️ 回滚方案（备用）

### 触发条件

| 情况 | 说明 |
|------|------|
| **A. 执行失败** | 命令报错或验证不通过 |
| **B. 功能异常** | 修改后预期外行为 |
| **C. 用户要求** | 用户主动要求回滚 |

### 回滚流程

```
回滚开始
    ↓
步骤 1：停止受影响服务
命令：pm2 stop [服务名]
    ↓
步骤 2：恢复备份（如有配置变更）
命令：cp [备份文件] [目标文件]
    ↓
步骤 3：重启服务
命令：pm2 restart [服务名]
    ↓
步骤 4：验证服务正常
命令：pm2 list && curl localhost:[端口]/health
    ↓
回滚完成
```

## ❌ 禁止事项

| 禁止 | 说明 |
|------|------|
| 禁止跳过根因分析 | 直接处理表象而不溯源 |
| 禁止跳过手册查阅 | 可能遗漏关键配置要求 |
| 禁止未评估影响就执行 | 可能引发连锁反应 |
| 禁止无回滚方案就执行 | 一旦失败无法恢复 |
| 禁止跳过用户审批 | 必须用户明确确认 |
| 禁止单模型决策 | 必须有交叉审核环节 |
