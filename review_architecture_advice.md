# 架构评审：AI助手任务状态管理方案

## 1. 问题理解准确性

**准确，但有重要遗漏：**

✅ **核心问题识别正确**：每日日志混杂已完成/未完成任务，导致新 session 启动时认知负担重，需要人工筛选。

⚠️ **遗漏点**：
1. **任务依赖关系**：某些任务可能有父子关系或阻塞关系，当前方案未考虑。
2. **任务上下文**：任务通常需要关联特定文件、对话或代码片段，纯文本列表可能丢失上下文。
3. **跨日任务追踪**：任务可能跨越多日，当前日志按日分割，缺乏连续性视图。
4. **任务优先级**：没有区分紧急/重要程度。
5. **系统复杂性**：AI助手本身是动态系统，频繁的文件读写可能影响性能。

## 2. 四个方案优劣分析

### 方案 A：活跃任务独立文件
**优点**：
- **简单直观**：单文件，易于读写和维护
- **分离关注点**：日志记录事件，任务文件管理状态
- **启动高效**：新 session 只需读取 ACTIVE_TASKS.md
- **版本友好**：Git 可追踪任务变更历史

**缺点**：
- **手动同步**：任务完成需要手动从文件移除，容易遗漏
- **无结构化**：纯文本格式难以自动解析、排序、筛选
- **并发风险**：多 session 同时修改可能冲突（但 AI 助手通常是单实例）
- **上下文丢失**：任务与原始日志记录分离

### 方案 B：每日日志内嵌状态标记
**优点**：
- **保持上下文**：任务状态与原始记录在一起
- **渐进实施**：可逐步在现有日志中添加标记
- **历史可查**：完成后仍保留完整记录

**缺点**：
- **过滤开销**：每次启动需要解析整个日志文件
- **标记一致性**：依赖人工标记，容易不一致
- **版本混淆**：同一个任务可能在不同日期多次出现
- **性能问题**：日志文件增长后解析成本增加

### 方案 C：结构化持久化任务表
**优点**：
- **强大查询**：可按状态、优先级、日期等灵活筛选
- **自动追踪**：可添加创建时间、完成时间、负责人等元数据
- **关系建模**：支持任务依赖、标签分类
- **并发安全**：数据库事务保证一致性

**缺点**：
- **过度工程**：为个人助手引入 PostgreSQL 过重
- **运维负担**：需要维护数据库、连接、备份
- **迁移成本**：现有日志需要导入
- **复杂度**：查询需要 SQL 或 ORM 知识

### 方案 D：定期清理快照
**优点**：
- **保持精简**：日志文件不会无限增长
- **历史归档**：已完成任务仍可查阅
- **简单实施**：只需定期脚本或手动操作

**缺点**：
- **信息丢失**：归档后可能难以查找
- **临时需求**：有时需要回顾较久之前的任务
- **手动维护**：需要定期执行清理
- **不解决核心问题**：只是缓解，未解决状态分离

## 3. 是否有更好的方案？

### 推荐方案：A+B 增强版 + 轻量级自动化

**核心思想**：保持文件系统的简单性，但增加适量自动化：

1. **ACTIVE_TASKS.md 结构化增强**：
   ```markdown
   ## [P1] 网关服务开发 (gateway)
   - 状态: TODO
   - 创建: 2026-04-05
   - 最后更新: 2026-04-05
   - 上下文: memory/2026-04-05.md#gateway
   - 标签: #tiandao #backend
   
   ## [P2] Admin 前端界面 (admin-app)
   - 状态: BLOCKED
   - 阻塞: 等待 gateway 完成
   - 创建: 2026-04-05
   - 上下文: memory/2026-04-05.md#admin-app
   - 标签: #tiandao #frontend
   ```

2. **自动化状态同步**：
   - 开发简单脚本，检测日志中的 `[DONE]` 标记，自动从 ACTIVE_TASKS.md 移除对应任务
   - 或添加 AI 助手命令：`/task-done <任务ID>` 自动更新

3. **每日日志预处理**：
   - 新 session 启动时，自动扫描前3天日志，提取未完成任务到临时视图
   - 保持 ACTIVE_TASKS.md 为权威来源

4. **任务发现助手**：
   - 实现简单启发式规则：日志中出现的 "TODO"、"待处理"、"接下来" 等关键词自动提示添加到任务列表

5. **轻量级数据库替代**：使用 SQLite 文件（无服务）存储任务，平衡结构化和简单性

## 4. A+B 结合是否合理？

**合理，但需改进**：

### 当前 A+B 方案的潜在问题：
1. **双重维护**：既要在日志标记状态，又要维护独立任务文件
2. **同步负担**：容易产生不一致
3. **认知分裂**：开发者在两个地方查看任务信息

### 改进建议：**主从架构**

- **主**：`ACTIVE_TASKS.md` 作为单一事实来源（Single Source of Truth）
- **从**：每日日志中的任务标记作为**引用**而非状态记录

**工作流**：
1. 发现新任务 → 直接添加到 `ACTIVE_TASKS.md`
2. 日志中只需简要提及任务，并引用任务ID：`[任务: gateway-dev]`
3. 完成任务 → 在 `ACTIVE_TASKS.md` 标记为 DONE，可保留完成时间和简要说明
4. 定期清理：每月将已完成任务移动到 `tasks/archive/YYYY-MM.md`

### 新 session 启动流程优化：
```
1. 读取 SOUL.md + USER.md（身份）
2. 读取 MEMORY.md（长期记忆）
3. 读取 ACTIVE_TASKS.md（当前任务）
4. （可选）快速扫描最近3天日志的变更部分
```

## 5. 实施建议

### 短期（立即）：
1. 创建 `ACTIVE_TASKS.md` 文件，从当前日志提取未完成任务
2. 定义简单标记格式（状态、优先级、上下文链接）
3. 更新 AGENTS.md 中的 session 启动流程

### 中期（1周内）：
1. 开发简单任务同步脚本
2. 添加任务管理技能（SKILL.md）
3. 在日志中建立任务引用规范

### 长期（可选）：
1. 实现任务自动发现（关键词检测）
2. 添加任务依赖和标签系统
3. 开发任务看板可视化（通过 Canvas 工具）

## 6. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 任务遗漏 | 定期（每天）检查日志 vs 任务文件一致性 |
| 状态不一致 | 任务完成后，脚本自动在日志中添加完成标记 |
| 文件冲突 | 使用文件锁或单 session 假设（AI 助手通常是单实例） |
| 过度自动化 | 保持手动核心，自动化仅辅助；避免复杂状态机 |

## 7. 结论

**推荐采用增强版 A+B 方案**：
- 以 `ACTIVE_TASKS.md` 为任务权威来源
- 每日日志保持事件记录性质，仅引用任务ID
- 添加轻量级自动化同步
- 保持系统简单、可维护、符合现有工作流

**关键成功因素**：
1. **最小化变更**：不颠覆现有日志系统
2. **渐进采纳**：可先手工维护，后添加自动化
3. **实用主义**：优先解决核心痛点（启动时任务筛选），而非构建完美系统

**最终提醒**：最适合的架构是那个能被实际使用和持续维护的架构。从简单开始，根据需要演化。
