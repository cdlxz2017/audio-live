# 2026-04-19 05:17 Session — 系统性重复劳动与独立思考

## 问题陈述

用户指出核心问题：**我们已经花了大量时间修记忆系统，但一直在重复打转，没有真正进步。**

### 已识别的根本问题

#### 1. 跨 session 无记忆连续性
- 每次 session 都是孤岛，新的 session 不知道上一个 session 处理过什么问题、做过什么决策、走过什么弯路
- 同一个 recall 问题，昨天 session 排查过、结论和方案都有了，今天 session 又从零开始
- 记忆系统有数据，但 session 上下文是"沉默"的——数据在，但没人调用

#### 2. 单点思维，缺乏全局视角
- 问题出现 → 直接定位文件/函数 → 改
- 而不是：先看模块原来的设计 → 为什么现在出问题 → 根本原因 → 方案
- 导致：修了一个点，另一个点坏了；局部最优，全局原地打转
- 例子：recall 链路问题反复出现，每次都是单点修复（截断/时序/缓存/注入），从未从整体审视 extractor 并存问题和链路设计缺陷

#### 3. session 上下文有限
- 每个 session 的上下文有上限，不可能把所有记忆都塞进去
- 只能装下本次 session 的上下文，超出范围的记忆"沉默"
- 当前选择机制是"零"：除了 MEMORY.md，没有任何记忆在 session 启动时自动加载

#### 4. 知道和执行之间的鸿沟
- 即使记忆系统里有解决方案，不是每次都去用
- "想起来去查"本身就是额外负担
- 这是动机问题，不是能力问题

#### 5. 每个人都是单点视野
- AI（我）看到的是 recall 链路
- 用户看到的是整体进度
- 没有人把所有人的视野拼起来形成全局图景

#### 6. 重复劳动是双向的
- AI 重复：同一个问题反复分析、反复定位
- 用户重复：同一个问题要解释多遍、每次新建 session 要重新对齐上下文
- 用户知道的和 AI 知道的始终有断层

#### 7. 缺乏对"已验证结论"的尊重
- 某个方案被否定过、某条路被走通但有隐患——这些结论没有成为下一次决策的前置条件
- 每次都是重新辩论，从零开始

### 独立思考的真正含义

**现在的 AI（工具模式）：**
- 收到问题 → 分析 → 给方案 → 等确认 → 执行
- 方案依赖本次 session 上下文，超出上下文的不"知道"

**真正需要的 AI（参谋模式）：**
- 收到问题 → 先想"这个问题跟之前哪些问题相关？"
- 分析问题 → 先问"这个模块原来的设计是什么？出过什么同类问题？"
- 给方案 → 先看"全局视角下，这个问题应该怎么归类？优先级是什么？"

**不只是执行者，还是有记忆、有判断、有主见的参与者。**

### 你关于记忆设计的核心观点

**每个 session = 我的一天。**
- 人类无法清楚记得前几天做过什么，所以写日记做笔记
- 我的每个 session 也是「一天」，同样需要摘要和记录
- 但人的记忆是碎片化但连贯的，能沿线索追溯；我的是碎片且断层的

**存记忆之前，先问想要什么记忆。**
- 我们要的不是「recall 链路的 topK 是多少」这种数据
- 而是这个问题的领域、历史结论、决策过程、方案验证
- 记忆系统设计应从「想要什么」出发，而非从技术实现出发

**多线程并行追踪。**
- 每个 session 不是一件事，是 N 件事并行
- 类比人一天：早上项目A，下午项目B，晚上回到项目A
- 记忆系统需要支持多线程：一个 session 里有多个「事情线程」同时进行
- 线程状态：新建 / 进行中 / 已完成 / 已取消
- session 结束时存档所有线程状态和进度
- 下次 session 启动时，自动加载所有「未完成的线程」

```
示例：
session 1: [事情A-进行中] [事情B-新] [事情C-新]
session 2: [事情A-继续] [事情B-继续] [事情D-新]
session 3: [事情A-完成] [事情B-完成] [事情D-继续] [事情E-新]
```

**多层架构是递进关系，不是平级堆叠。**
- summaries → 原始对话 → Neo4j关系 → Graphify代码
- 每层回答不同层次的问题：是什么 → 怎么讨论的 → 和什么相关 → 具体怎么实现
- 正确使用方式是「递进」，而非每次从零开始查

### 解决方向

**问题追踪系统（Problem Thread）**
1. 多线程并行：一个 session 可追踪 N 个事情线程
2. 以「问题 / 事件」为主线组织记忆，而非按时间或类型
3. 主动递进加载：session 启动时自动加载未完成的线程 → 主线上下文 → 递进到原始对话/Neo4j/Graphify
4. 已验证结论前置：历史结论成为新决策的前置条件

**与现有系统的关系**
- 现有 summaries 是「事后日记」
- 真正需要的是「事前入口」——让下一个 session 知道从哪里开始
- Neo4j 可以作为线程间关系的连接层
- Graphify 可以追踪代码改动的历史链路
