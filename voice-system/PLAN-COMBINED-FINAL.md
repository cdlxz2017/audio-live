## 二、共识：通讯录是基石

**两方案完全一致：通讯录是第一优先级。**

为什么？

```
没有通讯录：
  来电 → "13980819087 打来"
  → 你不知道是谁 → 推送没价值 → 功能天花板

有了通讯录：
  来电 → "老妈打来电话"
  → 你知道是谁 → 差异化处理 → AI 能力释放
```

### 通讯录能干的事

| 场景 | 效果 |
|------|------|
| 来电推送 | "📞 老妈打来电话" 而非 "139xxxx" |
| 黑白名单 | 骚扰自动挂断，VIP 秒接 |
| 按人聚合 | "本月与老妈通话 5 次，共 23 分钟" |
| 自然语言拨号 | "给老妈打电话" → 自动查号拨打 |
| 未回拨 | "3个未接：老妈/张三/快递" |
| 未知号码 | 自动记录，3次提醒后建议加入通讯录 |

### 通讯录数据结构（一张表起步）

| 字段 | 作用 |
|------|------|
| 姓名 | 必填，来电显示用 |
| 手机号 | 唯一标识 |
| 关系 | 家人/同事/朋友/快递/客服 |
| 重要程度 | 0普通 / 1重要 / 2紧急 |
| 黑名单 | 1 = 自动挂断 |
| 白名单 | 1 = 跳过振铃直接接 |
| 最后通话时间 | 自动更新 |
| 累计通话次数 | 自动累加 |

**号码匹配规则**：去 +86、去空格、去分隔符，匹配时做后7位模糊匹配。

## 三、共识：架构需要解耦

### 现在的问题

**一个进程干所有事：**

```
来电监听 → 录音 → ASR转写(20-60秒!) → LLM摘要 → 微信推送
          ↑ 这一步阻塞期间，串口缓冲区可能溢出，漏接来电
```

### 解耦后

**两个进程，各司其职：**

```
进程1：监听器（毫秒级响应）
  接电话 → 录音 → 写事件文件 → 返回监听

进程2：后处理器（异步，不阻塞）
  读事件 → 转写 → 摘要 → 推送 → 存数据库
```

**效果**：后处理再慢也不影响接听下一通电话。

## 四、风险清单：11 个问题，逐个解决

### 🔴 已验证发生（4个）

**R1：来电只显示号码**
- 根因：无通讯录
- 解法：Phase 1 建 contacts 表，来电查表

**R2：录音质量差（clipping + 电流声）**
- 根因：① 录音增益过高 ② 地环路噪声 ③ 模拟信号质量不稳
- 解法（三层）：
  - 立即：降低增益 + 后处理标准化
  - 硬件：音频隔离器到货后安装
  - 监控：每次录音自动检测音量峰值

**R3：ASR 阻塞来电监听**
- 根因：监听和处理在同一进程同一线程
- 解法：拆成两个进程（见第三节）

**R4：只能录到对方声音**
- 根因：硬件环回物理限制
- 解法：pactl 创建系统音频回环，合并双方声音（纯软件，0元）

### 🟡 潜在风险（5个）

**R5：TTS 外呼音频送不进模块**
- 解法：反向硬件环回（主机输出 → 音频线 → 模块 MIC 输入），成本 ¥5

**R6：串口断开无法自动恢复**
- 根因：USB 总线复位后 pyserial 抛异常
- 解法：systemd 替代 PM2（自动重启）+ 重连后检查残留通话

**R7：FunASR 断网时不可用**
- 解法：已有 Whisper 本地回退

**R8：SQLite 并发写入**
- 解法：WAL 模式允许读写并发

**R9：号码格式不统一导致匹配失败**
- 解法：标准化函数 + 后7位模糊匹配

### 🟢 小问题（2个）

**R10：API Key 散落在多个脚本中** → 统一 settings.yaml

**R11：录音文件无限增长** → cron 每周清理，保留30天

### Phase 2：录音质量修复（1-2天）

### Phase 3：监听-处理解耦（2-3天）

**里程碑**：后处理不再阻塞来电监听 ✅

## 六、关键决策（5个）

| 问题 | 选什么 | 为什么 |
|------|--------|--------|
| 进程管理器 | 先用 PM2，后面评估 systemd | 现有 PM2 在运行，改动最小 |
| 双向录音 | pactl loopback | 纯软件，0元 |
| TTS 外呼 | 反向硬件环回 | 上传不可用，这是唯一方案 |
| ASR 引擎 | FunASR 优先，Whisper 兜底 | FunASR 中文质量更好 |
| 解耦时机 | Phase 3 再做 | 先做通讯录和录音修复，收益更大 |
