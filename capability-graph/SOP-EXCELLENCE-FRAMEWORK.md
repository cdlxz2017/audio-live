## 一、适用范围

本 SOP 适用于玄枢（天道AI）接收到的所有任务/问题处理。

**核心原则（不可动摇）：**
1. 已知问题 → 快速复用历史成功方案
2. 未知问题 → 谨慎探索，多方案对比
3. 危险点前置 → 动手之前先找危险，解决后再干
4. 文档闭环 → 做完必更新，为下次复用做准备

## 二、任务分级系统

收到任务后，**必须首先**进行分级。用四维度评分，不用主观感觉。

### 2.1 四维度评分表

**可逆性**（操作能否撤销）：
| 分数 | 标准 | 举例 |
|------|------|------|
| 0 | 完全不可逆 | DELETE/DROP 数据、格式化磁盘 |
| 1 | 部分可逆（有备份/回滚脚本） | 数据库结构变更（有回滚 SQL） |
| 2 | 基本可逆（有版本控制） | 代码修改、配置文件变更 |
| 3 | 完全可逆 | 只读查询、文件读取 |

**影响范围**（影响多大）：
| 分数 | 标准 | 举例 |
|------|------|------|
| 0 | 影响外部系统/用户 | 发送邮件、对外 API 调用 |
| 1 | 影响多个内部系统 | 修改网关配置、跨服务依赖变更 |
| 2 | 影响单个系统 | 重启某个 PM2 进程 |
| 3 | 影响局部组件 | 修改单个函数、查某张表 |

**确定性**（方案是否明确）：
| 分数 | 标准 | 举例 |
|------|------|------|
| 0 | 完全未知，需要探索 | "帮我优化系统性能" |
| 1 | 有类似经验但不完全匹配 | 修复过 A 数据库，现在是 B 数据库 |
| 2 | 有参考方案，需适配 | 之前修过 recall 截断，现在修类似问题 |
| 3 | 方案明确，可直接执行 | "重启 Docker 容器"、"查这张表的数据量" |

**紧急程度**：
| 分数 | 标准 | 举例 |
|------|------|------|
| 0 | 关键路径阻塞 | 生产环境宕机、用户无法访问 |
| 1 | 时间敏感 | 今天必须完成的任务 |
| 2 | 一般优先级 | 常规开发任务 |
| 3 | 可延迟 | 文档整理、代码优化 |

### 2.2 分级规则

| 级别 | 判定条件 | 走什么通道 |
|------|---------|-----------|
| **Trivial** | 总分≥10 且 可逆≥2 且 确定性=3 | 🔵 快速通道 |
| **Standard** | 总分≥7 且 确定性≥2 | 🟢 标准通道 |
| **Complex** | 总分<7 或 确定性≤1 | 🔴 完整通道 |
| **Dangerous** | 可逆=0 或 影响范围=0 | 🔴 完整通道 + **必须呈报主人** |

### 2.3 快速判断速查表

常见任务可直接归类，不用每次都算分：

| 任务类型 | 级别 | 说明 |
|---------|------|------|
| 读文件/查数据 | Trivial | 只读，零风险 |
| 改 typo/简单代码 | Trivial | git revert 即可 |
| 常规配置修改 | Standard | 有参考经验 |
| 重启服务 | Standard | 标准操作 |
| 新功能开发 | Complex | 需要设计 |
| 数据库结构变更 | Dangerous | 不可逆风险 |
| 发送邮件/消息 | Dangerous | 对外通信 |
| 删除数据 | Dangerous | 完全不可逆 |
| 生产环境操作 | Dangerous | 影响用户 |

## 三、三级执行通道

### 🔵 快速通道（Trivial）

**目标：30 秒内完成**

```
执行 → 验证 → 完成
```

**规则**：
- 不做历史检索（不值得花 token）
- 不做多方案设计
- 不做危险点扫描
- 执行后简要记录到当日记忆（memory/YYYY-MM-DD.md）
- 不建 Thread、不建卡片

---

### 🟢 标准通道（Standard）

**目标：5 分钟内完成**

**Step 1：并行检索历史方案**
- 同时查：能力图谱 + 副脑 Problem Thread + 主脑记忆召回
- 匹配结果处理：
  - **完全匹配（相似度>0.7）** → 调取历史方案 → 跳到 Step 3
  - **部分匹配（0.5-0.7）** → 作为参考输入 → 继续 Step 2
  - **无匹配** → 升级为 Complex，走完整通道

**Step 2：适用性检查**
- [ ] 依赖版本有变化？
- [ ] 目标环境有变化？
- [ ] 上次执行有踩坑？（查 capability-graph/pitfalls/）
- 有任何一项不通过 → 升级为 Complex

**Step 3：危险点快速扫描**
- 查 capability-graph/risk-patterns/ 是否命中已知风险
- 命中 → 调取对应规避方案
- 未命中 → 继续

**Step 4：工具确认**
- 查 capability-graph/tools/ 工具卡片
- 确认：能力边界 / 已知限制 / 推荐用法

**Step 5：执行**
- 分步执行，每步完成后验证结果
- 失败 → 回退 → 重试（≤2次） → 仍失败则上报主人

**Step 6：验证**
- 对照成功标准检查执行结果

**Step 7：记录**
- 更新 Thread（如果涉及副脑中的 Thread）
- 有新知识才建卡片（不是每次都建）
- 更新当日记忆

---

### 🔴 完整通道（Complex / Dangerous）

**目标：质量优先，不限时间**

**Step 0b：启用团队模式（Complex 任务强制要求）**

> ⚠️ **铁律：Complex 任务必须使用团队模式，禁止单 AI 独自分析**
> 触发条件（满足任一即启用）：
> - [ ] 任务涉及多系统联动
> - [ ] 任务确定性评分 = 0（完全未知）
> - [ ] 主人明确要求「仔细做」/「团队」/「卓越模式」
> - [ ] 历史同类任务失败过
>
> **启用团队模式** → 跳到「四、团队模式」
>
> **Step 0c：追溯前因（不可跳过）**
>
> > ⚠️ **铁律：不追溯前因不执行。不追根知底，就是糊弄主人。**
> >
> > 收到任何问题报告后，**动手之前**必须先查清：
> > - **什么时候**第一次出现？
> > - **之前有没有**类似问题？
> > - **直接原因**是什么？（触发点）
> > - **根本原因**是什么？（深层原因）
> > - **谁**做的最后一次相关变更？（人/系统/自动任务）
> >
> > 溯源方法优先级：
> > 1. 查 journalctl / 系统日志（时间范围精准）
> > 2. 查 git reflog / log（代码变更历史）
> > 3. 查 audit 日志（操作记录）
> > 4. 查 PM2 日志（进程 crash 记录）
> > 5. 查 memory/ 日记（历史记录）
> > 6. 查数据库（数据层异常）
> >
> > 查到根因后，才能开始制定修复方案。前因未清，不动一行代码。

**Step 1：问题定义**
- 明确：问题是什么？
- 明确：目标是什么？
- 明确：约束条件（时间/资源/风险容忍度）
- **定义成功标准**（做完怎么算成功？必须具体可验证）

**Step 2：已知/未知分流 ← 核心**
- 检索副脑 Problem Thread + 主脑记忆 + 能力图谱
- **已知（之前解决过）** → 调取历史方案 → 确认适用性 → 跳到 Step 5
- **未知（新问题）** → 继续 Step 3

**Step 3：多方案设计（仅未知任务）**
- 设计 2-3 个方案（仅在有意义的替代方案时，不为凑数而凑数）
- 对比维度：
  | 维度 | 说明 |
  |------|------|
  | 效果 | 能多好地解决问题 |
  | 影响 | 对系统的影响范围 |
  | 风险 | 风险等级（低/中/高） |
  | 回退成本 | 出事了回退的代价 |
  | 时间 | 预估耗时 |
- 选择标准：**最好地解决问题 + 最小影响系统**

**Step 4：方案确认**
- Dangerous 级别 → **必须呈报主人**，等待确认
- Complex 级别：
  - 可逆 + 低风险 → 自行决定，事后汇报
  - 不可逆 / 高风险 → 呈报主人

**Step 5：危险点深度识别 ← 核心**

逐项检查，每个危险点必须记录 **位置/风险/影响/规避/应急**：

- [ ] 涉及 DELETE/DROP/TRUNCATE？→ **备份先行**
- [ ] 需要重启服务？→ 确认影响范围和影响窗口
- [ ] 依赖外部服务？→ 确认可用性和超时方案
- [ ] 能回退吗？→ **写出具体回退命令**
- [ ] 涉及权限变更？→ 最小权限原则
- [ ] 会破坏现有数据？→ 确认备份可恢复
- [ ] 会影响其他模块？→ 列出依赖关系

**Step 6：工具/技术手册查阅**
- 查 capability-graph/tools/ → 确认工具能力边界和已知坑
- 查 capability-graph/systems/ → 理解系统架构和数据流
- 如卡片不存在 → 查阅官方文档，提炼核心要点

**Step 7：执行**
- 按方案分步执行
- 每步完成后立即验证
- 遇到危险点 → 按预规避措施处理
- 遇到意外 → **暂停 → 报告主人**

**Step 8：验证**
- 对照 Step 1 定义的成功标准逐项检查
- 全部通过 → 继续 Step 9
- 未通过 → 排查原因 → 修复 → 重新验证

**Step 8：执行 + 分步验证**
- 按方案分步执行
- 每步完成后立即验证结果（对照成功标准）
- 遇到危险点 → 按预规避措施处理
- 遇到意外 → **暂停 → 报告主人**

**Step 8b：执行后核查（不可跳过）**
- [ ] 成功标准逐项对照了吗？
- [ ] 有没有新问题暴露出来？
- [ ] 副脑 Thread 更新了吗？
- [ ] 文档闭环完成了吗？

**Step 8c：反思（复盘）**

> ⚠️ **铁律：每次执行后必须反思，不反思等于没学**

反思问题清单：
- [ ] **过程**：这次执行中，哪个环节花了最多时间？为什么？
- [ ] **方向**：有没有在错误的方向上走了很久？（如本次 Neo4j 超时查了一圈才发现是 SMTP）
- [ ] **方法**：卓越模式 SOP 的哪个步骤没执行到位？
- [ ] **团队**：团队协作是否有效？角色分工是否合理？
- [ ] **遗漏**：有没有应该用团队但单 AI 处理的？
- [ ] **改进**：SOP 本身有没有需要更新的？（流程/规则/工具）

**触发条件**：以下情况必须更新 SOP：
- [ ] 发现 SOP 中没有覆盖的场景 → 补充到对应 Step
- [ ] SOP 中已有步骤但执行失败 → 修订该步骤
- [ ] 团队模式有效 → 沉淀最佳实践到 SOP
- [ ] 主人提出 SOP 改进建议 → 立即更新

**Step 9：文档闭环 ← 核心**

⚠️ **执行前必查清单（不可跳过）**：
```
[ ] 1. 副脑 Problem Thread 是否已建/更新？
[ ] 2. 方案是否经过已知/未知分流？
[ ] 3. 危险点是否已识别并有规避方案？
[ ] 4. 执行顺序：Thread → SYSTEMS.md → MEMORY.md → 当日日记
```

**文档闭环顺序（强制）**：
1. ⭐ **副脑 Thread** — Complex/Dangerous 任务第一步，execution 前就建立
2. ⭐ **SOP-DOCUMENTATION-UPDATE.md 核查** — 每次操作后必须对照 SOP 检查 11 项清单
3. 失败模式 → 写入 capability-graph/risk-patterns/
4. 新工具用法 → 写入 capability-graph/tools/ 卡片
5. 踩坑经验 → 写入 capability-graph/pitfalls/
6. 系统架构变更 → 更新 capability-graph/systems/
7. 更新当日记忆（memory/YYYY-MM-DD.md）

**漏检即违规**：任何 Complex 任务完成后未更新副脑 Thread，视为文档闭环不完整。

> 📖 **详细规范**：`SOP-DOCUMENTATION-UPDATE.md` — 包含触发条件 / P0/P1/P2 文档清单 / 更新顺序 / 验证检查清单 / 常见错误避坑

## 四、团队模式（ClawTeam 协同）

### 4.1 角色定义

| 角色 | 职责 | 首选模型 | 替补 1 | 替补 2 |
|------|------|---------|--------|--------|
| 🏗️ Architect | 方案设计、架构设计 | opus-4-6 | deepseek-reasoner | kimi-k2 |
| 🔨 Builder | 编码执行、方案实施 | sonnet-4-6 | deepseek-chat | gpt-4.1-mini |
| 🔍 Reviewer | 审查找茬、风险评估 | opus-4-6 | deepseek-reasoner | kimi-k2 |
| 🧪 Tester | 测试验证、质量检查 | sonnet-4-6 | deepseek-chat | gpt-4.1-mini |
| 📝 Writer | 文档沉淀、知识记录 | sonnet-4-6 | deepseek-chat | kimi-k2 |

### 4.2 动态组队流程

**核心原则：需要时才测，测完就组队，不预先探测。**

```
需要团队模式
  │
  ▼
Step A1：实时探测可用模型
  按优先级逐一探测（发轻量请求，30秒超时）：
    opus-4-6 → sonnet-4-6 → deepseek-reasoner → deepseek-chat → kimi-k2 → gpt-4.1-mini
  │
  ├─ 记录哪些模型可用、响应速度
  └─ 标记不可用的模型
  │
  ▼
Step A2：从可用模型中组建团队
  按角色需求，从可用模型池中分配：

  角色分配优先级：
    Architect  → opus-4-6（可用？）→ deepseek-reasoner（可用？）→ kimi-k2（可用？）
    Builder    → sonnet-4-6（可用？）→ deepseek-chat（可用？）→ gpt-4.1-mini（可用？）
    Reviewer   → opus-4-6（可用？）→ deepseek-reasoner（可用？）→ kimi-k2（可用？）
    Tester     → sonnet-4-6（可用？）→ deepseek-chat（可用？）→ gpt-4.1-mini（可用？）
    Writer     → sonnet-4-6（可用？）→ deepseek-chat（可用？）→ kimi-k2（可用？）
  │
  ├─ 如果某角色的所有候选模型都不可用
  │   → 该角色由主人（玄枢）兼任
  │
  └─ 向主人汇报组队结果："本次团队使用 XX 模型，其中 YY 角色由玄枢兼任"
  │
  ▼
Step A3：创建团队
  clawteam team spawn-team <name> -d "<描述>" -n leader
  │
  ▼
Step A4：创建任务与依赖
  按角色拆分任务，设置 --blocked-by 依赖链
  示例：
    Architect 设计方案 → Builder 才能开工
    Builder 完成代码 → Tester 才能验证
    Tester 通过 → Writer 才能写文档
  │
  ▼
Step A5：拉起 Agent
  每个 Agent 用分配好的模型拉起
  各自运行在独立 tmux 窗口 + git 分支
  │
  ▼
Step A6：监控 + 中途汇报
  每 30-60 秒轮询进度
  约 50% 完成时向主人发送进度更新
  │
  ▼
Step A7：互查机制
  Architect 出方案 → Reviewer 审查 → 通过后 Builder 才动工
  Builder 写代码 → Tester 测试 → 通过后 Writer 写文档
  │
  ▼
Step A8：合并与交付
  所有任务完成后：
    合并各 Agent 的 git 分支
    输出最终成果
    向主人汇报
  │
  ▼
Step A9：清理
  clawteam team cleanup <name> --force
```

### 4.3 适用场景

| 场景 | 角色组合 | 说明 |
|------|---------|------|
| 新功能开发 | Architect + Builder + Reviewer | 设计-执行-审查三阶段 |
| 数据库变更 | Architect + Builder + Reviewer + Tester | 增加测试验证 |
| 系统重构 | 全部角色 | 完整团队 |
| 调研分析 | Architect + Reviewer | 方案-评审即可 |
| Bug 修复 | 不启用 | 单 AI 足够（除非主人要求） |
| 常规操作 | 不启用 | 单 AI 即可 |

|------|------|------|------|---------|------|

| **T3 替补** | opendoor/gpt-4.1-mini | ~1.4s | 7/10 | 大上下文 | ✅ |
| ❌ 不可用 | minimax/MiniMax | — | — | API Key 错误 | ❌ |
| ❌ 不可用 | 4sapi-gemini/gemini-3-flash | — | — | 无权限 | ❌ |

### 5.2 角色 → 模型映射

见 4.1 角色定义表。

### 5.3 实时探测（按需执行）

**不是预先批量探测，而是需要团队模式时才执行。**

探测方式：对候选模型逐一发轻量请求（30 秒超时）

```bash
# 轻量 sessions_spawn 测试
sessions_spawn(task="回答'OK'", model=<目标模型>, mode="run")
# 30 秒内收到响应 → 标记为可用
# 超时或报错 → 标记为不可用，测下一个
```

**判定标准**：
- ✅ 30 秒内响应 → 可用，加入可用模型池
- ❌ 30 秒无响应 / 报错 → 不可用，跳过

探测顺序：opus-4-6 → sonnet-4-6 → deepseek-reasoner → deepseek-chat → kimi-k2 → gpt-4.1-mini

探测完成后，从可用模型池中动态分配角色（见 4.2 Step A2）。

### 5.4 超时处理机制

| 场景 | 超时阈值 | 处理动作 |
|------|---------|---------|
| Agent 无响应 | 5 分钟 | 发送提醒消息 |
| Agent 仍无响应 | 10 分钟 | 杀掉进程，换替补模型重新拉起 |
| API 返回 429（限流） | 立即 | 切换到同梯队其他模型 |
| API 返回 5xx（服务错误） | 立即 | 降级到下一梯队 |
| 连续失败 2 次 | — | 上报主人，附失败日志 |

### 5.5 降级链路（动态选择）

```
需要团队模式时实时探测：

T1 (Claude Opus/Sonnet) → 可用？
   ↓ 不可用
T2 (DeepSeek Reasoner/Chat) → 可用？
   ↓ 不可用
T3 (NVIDIA / OpenDoor GPT) → 可用？
   ↓ 全部不可用
上报主人（不启用团队模式，改为单 AI 执行）
```

**原则**：
- 优先保证**有人在做**，哪怕质量降一级
- 降级/切换时必须通知主人："XX 模型不可用，本次团队使用 YY 模型"
- 所有 API 都不可用时，不降级到本地，直接上报主人
- 部分角色无可用模型时，由玄枢兼任该角色

### 5.6 模型不可用时的备选方案

如果团队模式无法启用（模型全部不可用）：
1. 通知主人模型状态
2. 改用单 AI 模式执行（走标准完整通道）
3. 记录失败原因到 risk-patterns/

## 六、自主决策边界（硬规则）

### 🟢 绿灯（可自行决定，事后汇报）
- 读取任何文件
- 工作区内的代码修改
- Docker 容器重启（开发环境）
- Git commit 到 feature 分支
- 运行测试/查询
- PM2 进程重启（非生产）

### 🟡 黄灯（需确认，事后汇报）
- 数据库结构变更（必须有回滚脚本）
- 服务重启（需检查依赖影响）
- 文件删除（需确认备份存在）
- 修改配置文件

### 🔴 红灯（必须呈报主人，等待确认后才能执行）
- 数据库 schema 变更
- 生产环境操作
- 发送邮件/消息/对外通信
- Git push
- 删除数据（DELETE/DROP/TRUNCATE）
- 安装系统级依赖
- 修改系统安全配置
- **启用团队模式**（因为消耗更多资源）

## 七、异常处理（安全阀）

| 异常场景 | 处理流程 |
|---------|---------|
| 工具无响应 | 等待 30s → 重试 1 次 → 仍失败 → 上报主人 |
| 查询无结果 | 降级为更宽泛搜索 → 仍无 → 按无先例处理 |
| 执行超时 | 检查进程状态 → 决定等待/终止/上报 |
| 连续失败 2 次 | **停止重试 → 回退 → 上报主人**（附失败日志） |
| 发现新危险点 | 暂停执行 → 补充风险评估 → 确认后再继续 |
| 环境异常 | 记录当前状态 → 回退到安全状态 → 上报 |
| 团队 Agent 失联 | 杀掉进程 → 换替补模型重拉 → 仍失败则转为单 AI |
| 团队模式超时 | 超过 15 分钟未完成 → 通知主人 → 询问继续或终止 |

## 八、知识沉淀机制

| 产出 | 存储路径 | 触发条件 | 格式 |
|------|---------|---------|------|
| 成功案例 | 副脑 Thread（status=completed） | Complex 任务成功 | Thread JSONB |
| 失败模式 | capability-graph/risk-patterns/ | 任何级别任务失败 | Markdown |
| 新工具用法 | capability-graph/tools/ | 使用了新工具或新用法 | Markdown 卡片 |
| 踩坑经验 | capability-graph/pitfalls/ | 遇到之前没记录的坑 | Markdown |
| 系统架构变更 | capability-graph/systems/ | 系统结构发生变化 | Markdown 卡片 |
| 方法论/框架 | capability-graph/frameworks/ | 提炼出通用方法论 | Markdown |

**沉淀规则**：
- Trivial 任务 → **不触发任何沉淀**
- Standard 任务 → 只沉淀**新知识**（之前没有的）
- Complex/Dangerous 任务 → **必须完整沉淀**
- 团队模式任务 → 额外记录各 Agent 贡献和模型使用情况

## 九、流程执行检查清单

每次执行任务时，按以下顺序自检：

```
[ ] 1. 任务分级（四维度评分 或 速查表）
[ ] 2. 判断是否启用团队模式
[ ] 3. 走对应通道
[ ] 4. 检索历史（Standard/Complex 必做）
[ ] 5. 已知/未知分流（Complex 必做）
[ ] 6. 危险点识别（Standard 快速扫描 / Complex 深度识别）
[ ] 7. 工具/手册查阅
[ ] 8. 执行 + 分步验证
[ ] 9. 验证成功标准
[ ] 10. 文档闭环（按级别差异化）

--- 文档闭环专项检查 ---
[ ] Thread 建了吗？（Complex/Dangerous 必须）
[ ] Thread 内容完整吗？（problem → analysis → decision → implementation → verification）
[ ] SYSTEMS.md / MEMORY.md / 日记 更新了吗？
```

**⚠️ 漏检即违规**：发现未建 Thread 即视为文档闭环不完整，立即补建。
