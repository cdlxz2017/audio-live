# 文档更新 SOP — 标准操作流程

> **版本**: v1.1
> **创建时间**: 2026-04-20
> **目的**: 确保每次系统变更后，文档同步更新，不遗漏、不乱更新
> **强制要求**: 任何代码/配置/架构变更，必须触发文档更新检查

---

## 0. 执行规范（铁律，2026-04-20 新增）

> ⚠️ 本规范是对卓越模式 SOP 的补充，所有 Complex 任务必须遵守：

| 规则 | 说明 | 违反视为 |
|------|------|---------|
| **不追溯前因不执行** | 任何故障/问题，动手修之前必须先查清根因（时间线/触发点/深层原因）| 糊弄主人 |
| **卓越模式强制团队** | Complex 任务必须启动团队（Architect + Investigator 等），禁止单 AI 独自分析 | 质量事故 |
| **反思后更新 SOP** | 每次执行后必须复盘，发现 SOP 遗漏立即更新 | 持续改进 |
| **文档先查再写** | 更新文档前，先读现有内容，确保不覆盖有价值的历史信息 | 信息丢失 |

**执行顺序**：
```
收到 Complex 任务
  → Step 0b：启动团队模式（不是单 AI）
  → Step 0c：追溯前因（不追根知底不动手）
  → Step 1-9：正常执行
  → Step 8c：反思复盘
  → 更新相关文档
```

## 一、触发条件（必须检查文档）

只要发生以下任一情况，立即启动本 SOP：

| 触发条件 | 说明 |
|---------|------|
| 新建系统/功能 | 新增组件、服务、脚本 |
| 修改系统/功能 | 对现有系统的重大修改（接口、配置、架构） |
| 新增/修改 API Key | 凭证管理、系统配置 |
| 新增/修改数据库表 | schema 变更 |
| 新增/修改 cron/定时任务 | 调度变更 |
| 新增/修改 PM2 进程 | 进程管理变更 |
| 新增/修改 skill/script | 工具变更 |
| 安全加固/权限变更 | 权限、认证相关 |
| 新增子仓库/Git 提交 | 版本控制相关 |

## 二、文档清单（按优先级）

### P0 — 必须更新（影响所有操作者）

| 文件 | 路径 | 更新触发 | 更新内容 |
|------|------|---------|---------|
| **系统注册表** | `SYSTEMS.md` | 新增系统/功能/服务 | 新增章节（触发词/路径/状态） |
| **能力索引** | `MEMORY.md` 能力索引总表 | 新增能力/工具/系统 | 新增索引条目 |
| **今日日记** | `memory/YYYY-MM-DD.md` | 任何变更 | 追加操作记录（APPEND 模式） |

### P1 — 应该更新（影响维护者）

| 文件 | 路径 | 更新触发 | 更新内容 |
|------|------|---------|---------|
| **工具手册** | `TOOLS.md` | 新增工具/配置/凭证 | 新增工具说明 |
| **项目档案** | 项目目录下 `README.md` 或 `SYSTEM-REVIEW.md` | 项目有重大修改 | 更新状态/架构/变更记录 |
| **子仓库 CHANGELOG** | `projects/*/CHANGELOG.md` | 子仓库有提交 | 追加变更条目 |
| **Git 提交** | 主仓库 + 各子仓库 | 有新提交 | commit message + 关联 issue |

### P2 — 按需更新（影响开发者）

| 文件 | 路径 | 更新触发 | 更新内容 |
|------|------|---------|---------|
| **设计文档** | `memory/*-DESIGN.md` | 方案设计完成 | 状态改为「已实施」，补充实施细节 |
| **技术知识库** | `tech-knowledge/` | 新增技术概念/架构 | 向量录入 |
| **SOP 文档** | `SOP-*.md` | 流程固化/新 SOP | 新增或更新 SOP |
| **API 设计文档** | `memory/*API*.md` | API 变更 | 更新接口说明 |

## 三、更新决策树

```
变更发生
    │
    ├── 新增系统/服务/功能？
    │       └── → SYSTEMS.md（新增章节）
    │           → MEMORY.md（索引条目）
    │           → 项目 README（如果存在）
    │
    ├── 新增文件/script/工具？
    │       └── → TOOLS.md（新增工具说明）
    │           → SYSTEMS.md（如果触发词相关）
    │
    ├── 修改了配置文件/凭证/API Key？
    │       └── → credentials/ 对应 .env
    │           → API-KEY-MANAGEMENT.md（如果涉及密钥管理）
    │           → SYSTEMS.md（如果涉及系统服务）
    │
    ├── 新增 cron / PM2 进程？
    │       └── → SYSTEMS.md（相关系统章节）
    │           → cron 说明 / PM2 说明
    │
    ├── 新增数据库表/修改 schema？
    │       └── → MEMORY.md 数据链路图
    │           → 项目 ARCHITECTURE.md
    │
    ├── 安全/权限变更？
    │       └── → SECURITY.md（如存在）
    │           → ~/.config/security/（安全档案）
    │
    └── 新增子仓库提交？
            └── → memory/YYYY-MM-DD.md（追加记录）
                → 子仓库 CHANGELOG.md（如存在）
```

## 四、更新顺序（强制顺序）

每次变更后，按此顺序更新文档：

```
第一步：更新 SYSTEMS.md（核心注册表）
    ↓
第二步：更新 MEMORY.md（能力索引）
    ↓
第三步：更新 memory/YYYY-MM-DD.md（追加今日日记）
    ↓
第四步：更新 TOOLS.md（如涉及工具）
    ↓
第五步：更新项目专属文档（子仓库 README / CHANGELOG）
    ↓
第六步：更新设计文档状态（如有设计文档）
    ↓
第七步：Git 提交（先子仓库，再主仓库）
    ↓
第八步：验证（检查所有应更新的文件是否都已更新）
```

## 五、今日日记格式规范

`memory/YYYY-MM-DD.md` 采用 **APPEND 模式**，每次追加新章节：

```markdown
---

## [操作标题]（时间区间）

**背景**：为什么做这个操作

**变更内容**：
- ✅ [文件/组件A] — [简短说明]
- ✅ [文件/组件B] — [简短说明]

**Git 提交**：
- `memory-system` 子仓库：commit xxx — [简要说明]
- workspace 主仓库：commit xxx — [简要说明]

**验证结果**：
- [验证项1]：✅ 正常
- [验证项2]：✅ 正常

**注意事项**：
- [如果有任何特殊事项]
```

## 六、Git 提交规范

### 子仓库提交顺序

```
1. 先子仓库提交（memory-system / 其他子仓库）
2. 再主仓库提交（workspace）
```

### Commit Message 格式

```
<type>(<scope>): <subject>

<type>: feat | fix | docs | refactor | test | chore
<scope>: 影响的子系统（memory-system / audit / credentials 等）
<subject>: 简短描述（中文，不超过50字）
```

### 常见场景 Commit Message 示例

| 场景 | Commit Message |
|------|---------------|
| 新增审计系统 | `feat(audit): Phase1 审计系统核心基础设施` |
| 修改凭证 | `feat(credentials): 新增XX凭证到中央管理` |
| 文档更新 | `docs: 更新SYSTEMS.md - 新增审计系统章节` |
| Bug 修复 | `fix(hook): session-capture-hook 修复审计日志漏记` |

## 七、验证检查清单

每次操作完成后，逐项核对：

```
□ SYSTEMS.md — 相关章节已更新/新增？
□ MEMORY.md — 能力索引已更新？
□ memory/YYYY-MM-DD.md — 今日日记已追加？
□ TOOLS.md — 工具说明已更新（如果涉及工具）？
□ 项目文档 — 子仓库 README/CHANGELOG 已更新（如果涉及项目）？
□ 设计文档 — 设计文档状态已更新为「已实施」（如果已有设计）？
□ Git 提交 — 子仓库和主仓库都已提交？
□ 监控/健康检查 — 新增的系统是否已接入监控？
□ Cron — 新增的定时任务是否已加入 crontab？
□ 凭证 — 新增的密钥是否已写入 credentials/？
```

## 八、常见错误与避坑

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 只改代码不写日记 | 后续无法追溯 | 每次操作都追加到当日日记 |
| 只更新 SYSTEMS.md 没更新 MEMORY.md | 能力索引不完整 | 两个文件同步更新 |
| 修改了系统但没更新触发词 | 主人无法唤醒该功能 | SYSTEMS.md 触发词必须同步 |
| 子仓库改了没提交 | Git 历史不完整 | 先子仓库再主仓库 |
| 新增 cron 但没加到 crontab | 定时任务不执行 | 新增任务立即加入 crontab |
| 新增敏感配置但没加到 credentials/ | 密钥散落各处 | 凭证统一管理 |
| 只更新内存不更新文档 | 系统重构后文档过时 | 代码和文档永远是兄弟 |
